> 내부 프로젝트명: 불혹 (不惑)
> 정식 명칭: GRIDGE Wiring AI (줄여서 Wiring)
> 최종 업데이트: 2026.04.06
> 이 문서는 기획 논의 전체 맥락을 보존하기 위한 종합 문서입니다.

## 1. 제품 정의

### 1.1 한 줄 정의
개발 관련 모든 업무를 AI Agent 기반으로 관리하는 그룹웨어. 팀 단위 Claude Code / Cursor / Codex에 대한 관제 센터 역할이며, AI가 액션을 수행하고 HITL(Human-in-the-Loop) 과정에서 사람이 확인/승인하는 인터페이스.

### 1.2 핵심 경험 정의
**"HITL 노드에서 유저가 확인/승인하는 행위 자체가 곧 프로젝트 관리 행위"**

기존 PM 도구(Jira, Clickup)에서는 "관리"와 "실행"이 분리되어 있지만, Wiring에서는 HITL 승인이 곧 관리이자 실행 지시. 유저 입장에서 인박스를 비우는 행위 = 프로젝트가 전진하는 행위.

### 1.3 AiOPS와의 관계
- **현재**: 별도 트랙으로 진행
- **최종 목표**: AiOPS Logging System → Wiring의 모니터링/코칭 모듈로 흡수
- **공유 인프라**: user/org 모델, 로그 수집 인프라(API 프록시, DB 스키마), 인증/권한 체계(RBAC)
- **통합 영역**: AI 사용 로그 수집/분석 → Document/Skill 활용 데이터, 비용 관리 → BM Agent 데이터 소스, 역량 코칭 + 성숙도 → 유저 AI 활용 능력 모니터링

## 3. Context Layer

### 3.1 정의
"회사 틀"을 수집하고 실시간으로 동기화하는 살아있는 데이터 레이어. Wiring의 근간 인프라.

### 3.2 수집 소스 및 동기화 방식

| 소스 | 동기화 방식 | 수집 데이터 |
|------|-----------|-----------|
| Git | Webhook (push/merge/PR) | 코드 구조, 의존성, 컨벤션, 브랜치 전략, 커밋 컨벤션, 코드 리뷰 히스토리 |
| 문서 도구 (Notion, Confluence, Wiki) | API 폴링/Webhook | 기획서, PRD, 디자인 시스템 문서, 온보딩 문서, ADR |
| 커뮤니케이션 (Slack, Teams) | 채널 구독 | 기술 의사결정 발언 |
| CI/CD (GitHub Actions, Jenkins) | 파이프라인 변경 감지 | 배포 규칙, 테스트 커버리지 기준 |
| 디자인 (Figma) | Webhook | 디자인 시스템 컴포넌트 변경 |
| DB | 직접 연결/덤프 업로드 | 스키마, 테이블 구조, 데이터 분류 |

### 3.3 반영 방식 이원화

**Auto-sync (자동 반영)**
- 사실 기반 데이터: 코드 구조, 패키지 버전, API 명세, 파일 트리
- 변경 시 즉시 반영

**GM 제안 → HITL 승인**
- 해석이 필요한 데이터: 정책 변경, 기술 결정, 컨벤션 변경
- GM이 변경 감지 → "회사 틀에 반영할까요?" 제안 → 사람 승인 시 반영
- GM의 판단 기준: Context Layer에서 1차 학습, 2차로 유저에게 기준 허락 요청

### 3.4 Context Layer와 Skill의 관계
- Context Layer = "회사가 뭘 갖고 있는지" (코드, 문서, 정책, 인프라)
- Skill = "회사가 어떻게 일하는지" (패턴, 프로세스, 효과적인 방법)

## 4. 온보딩

### 4.1 최소 진입 (STEP 1) — 택1 이상
- **프로덕트 URL 입력** → 크롤링으로 UI 구조, 기능 맵, 사용 기술 스택 추론
- **소스코드 업로드** (zip 또는 Git URL) → 코드 분석으로 아키텍처, 패키지, 컨벤션, API 구조 추출

프로덕트 링크만으로 시작 가능 = **기존 프로덕트 리빌딩, 경쟁사 분석 용도로도 확장 가능** (의도된 설계).

GM + SM이 자동으로 초기 Context Layer 생성. 분석 결과를 보여주면서 "소스코드도 연결하면 더 정확해집니다" 유도.

### 4.2 초기 Context 확인 (STEP 2)
- AI 분석 결과를 사람에게 보여주고 확인 + 보충
- "이 프로젝트는 Next.js 14 + Supabase 기반이고, 모노레포 구조이며, REST API 패턴을 사용합니다" 요약

### 4.3 점진적 소스 연결 (STEP 3)
- Git 레포 → Notion/Confluence → Slack → Figma → CI/CD → DB
- 연결할수록 Context Layer 풍부 → Agent 정확도 상승
- STEP 3의 각 소스도 웹 in 웹으로 Main Workspace 내에서 접근 가능

## 5. 프로젝트 계층 구조

```
Project (프로젝트)
  목표, 기간, 참여 인원, Context Layer 범위
    ↓
Epic (상위 티켓)
  프로젝트 목표 달성을 위한 큰 단위 업무
    ↓
Ticket (하위 티켓)
  Agent가 실행 가능한 단위. Flowchart의 노드.
    ↓
Sub-task (서브 태스크)
  Agent 간 소통에서 자동 생성되는 세부 작업
```

### 예시
```
[Project] SNS 서비스 v2.0
├── [Epic] 이미지 공유 기능
│   ├── [Ticket] 이미지 업로드 API 개발
│   │   ├── [Sub] S3 버킷 설정
│   │   ├── [Sub] 리사이징 파이프라인
│   │   └── [Sub] API 테스트
│   ├── [Ticket] ImageUploader 컴포넌트
│   ├── [Ticket] 공유 링크 생성
│   └── [Ticket] 보안 검토
├── [Epic] 피드 알고리즘 개선
└── [Epic] 성능 최적화
```

### Epic 간 의존성
- PM이 자동 감지 → 유저에게 제안 → 유저 선택 또는 직접 설정
- Flowchart 편집과 동일한 3단계 패턴 (PM 제안 → 유저 수용/반박 → 직접 수정)
- Project 레벨에서 Epic 간 관계를 보는 상위 Flowchart 별도 존재 (줌 인/줌 아웃 전환)

## 6. AI Agent 시스템

### 6.1 기본 Agent (9종)

| 약어 | 역할 | 설명 |
|------|------|------|
| PM | Project Manager | 업무 분배, 가드레일, 티켓 발행, 일정 관리, Agent 고용, 소통 토폴로지 설정 |
| GM | Governance Manager | 회사 방향 일치 검토, 코드 의존성, 자료 선별, HITL 승인 감독, 보안 경계 관리 |
| SM | Spec Manager | 기술 수준 결정, 스펙/프레임워크 제안, 개발 방향 추천, **AI 모델 배분 제안** |
| Dsn | Designer | UI 디자인 수행, 유저 제안 이미지 분석 |
| Pln | Planner | 기획서 제작, 하위 티켓 생성, 정책 수립, Wireframe |
| FE | Front-end Developer | 프론트엔드 개발, 기술 스펙 제안, API 문서 업데이트 |
| BE | Back-end Developer | 백엔드 개발, 기술 스펙 제안, API 문서 생산 |
| BM | Budget Manager | 토큰/인력 자원 평가, 티켓 발행 시 비용 제안, ROI 비교 |
| HR | HR Manager | HITL 인력 시간 고려, 내부/외부 전문가 매칭 제안 |

### 6.2 추가 Agent
- GRIDGE Market / GitHub에서 검색하여 제안
- MCP 활성화된 Agent 연결 가능
- PM이 검토 후 제안
- 직접 Agent 추가하여 상시 활성화 가능 (MCP 연결)

### 6.3 Agent 소통 구조

#### PM 소통 토폴로지 설정
PM이 티켓 시작 시점에 소통 모드를 설정. 티켓마다 다를 수 있고, 진행 중 변경 가능.

- **중앙집중형** — PM 경유 필수. 복잡한 티켓, 의존성 많은 경우.
- **페어링형** — 특정 Agent 쌍 직접 소통 허용, 나머지는 PM 경유. ("SM↔FE 직접 소통해, 나머지는 나한테 보고")
- **자율형** — 전체 직접 소통, PM은 감독만. 단순하거나 맥락 충분한 경우.

PM이 판단하는 기준: 티켓 복잡도, 보안 등급, 관련 Agent 수, 이전 유사 티켓 경험.

#### 4단계 가시성

| 레벨 | 설명 | 보이는 곳 |
|------|------|---------|
| 레벨 1 | 결과만 — HITL 큐에 최종 결론만 도착 | HITL 큐 |
| 레벨 1.5 | 티켓 단위 — Agent 논의에서 결정된 activity가 서브 티켓으로 칸반에 반영 | 칸반보드 |
| 레벨 2 | 논의 요약 — Agent 간 논의 타임라인 요약본 | Flowchart 노드 클릭 |
| 레벨 3 | 전체 로그 — Agent 소통 원문 전체 | Document 섹션 |

칸반이 실시간으로 살아 움직임: 유저가 안 보는 사이에도 Agent가 작업하면서 서브 티켓 생성/상태 이동/완료. 유저가 칸반 열었을 때 현황 바로 보임.

칸반에서 유저 개입 가능:
- 서브 티켓 드래그로 우선순위 변경 → PM 일정 재조정
- 서브 티켓 삭제 → PM "영향 있습니다" 경고
- 서브 티켓 직접 추가 → PM이 받아서 Agent 할당

## 7. AI 모델 배분 (SM + BM 협업)

### 7.1 발생 시점
티켓(또는 Epic) 생성 → PM이 Agent 할당 → **SM이 각 Agent별 AI 모델 배분 제안 + BM이 예상 비용 산출** → HITL 큐에 "AI 모델 배분 확인" 도착

### 7.2 제안 내용
- Agent별 어떤 AI 모델(Claude Sonnet, GPT-4o, Gemini Flash, Claude Haiku 등)을 어떤 비율로 사용할지
- 각 모델별 예상 토큰 (입력/출력) + 예상 비용
- 전체 비용 요약 (도넛 차트)
- SM 코멘트 (왜 이렇게 배분했는지 근거)
- 비용 절감 대안 (현재 구성 vs 다운그레이드 옵션들, 품질 영향 표시)

### 7.3 유저 액션
- 제안대로 승인
- 대안 중 선택
- 모델 비율 슬라이더로 직접 조정 (실시간 비용 재계산)
- 승인 시 agentStore에 배분 정보 저장 → Agent 작업 시작

## 8. HITL 큐/인박스

### 8.1 핵심 UX 루프

```
큐/인박스에 HITL 요청 도착
    ↓
유저가 아이템 클릭
    ↓
화면 전환:
  TopBar → 액션 아이템 타이틀
  Main → 유형별 뷰 (Flowchart / 코드 diff / 문서 / Figma / 모델 배분 등)
  Chat → 해당 큐의 맥락 브리핑 (경위, Agent 간 논의 요약, 선택지)
    ↓
유저가 Chat에서 Agent와 티키타카하며 의사결정
    ↓
승인/반려/수정 지시
    ↓
Flowchart 동적 업데이트, 다음 노드로 진행
    ↓
큐/인박스에서 해당 아이템 완료 처리
```

### 8.2 HITL 유형 + Main 렌더링

| 유형 | 트리거 | Main 렌더링 |
|------|--------|------------|
| 의사결정 (spec_decision) | SM 스펙 제안, PM 일정 조정 | Flowchart + 선택지 패널 |
| 코드 리뷰 (code_review) | FE/BE 코드 작성 완료 | 웹 in 웹 코드 diff 뷰어 |
| 기획 검토 | Pln 기획서/와이어프레임 완료 | 웹 in 웹 문서 뷰어 |
| 디자인 검토 (design_review) | Dsn UI 시안 완료 | 웹 in 웹 Figma/이미지 뷰어 |
| 보안 승인 (security_approval) | GM 보안 등급 자료 접근 요청 | Flowchart + 승인 모달 |
| 비용 확인 (cost_approval) | BM 예산 초과 알림 | Flowchart + 비용 요약 패널 |
| 담당자 배정 (assignment) | HR 내부/외부 인력 매칭 | 후보 카드 + ROI 비교 |
| AI 모델 배분 (model_allocation) | SM + BM 모델 배분 제안 | Agent별 배분 바 + 도넛 차트 + 대안 + 슬라이더 |

### 8.3 HITL에서 유저한테 오는 것 3종류

**승인 요청 (Decision)**
- GM: "이 코드 모듈에 접근해야 하는데, 보안 등급이 높습니다. 허용할까요?"
- SM: "스펙 선택지 3개 중 하나를 골라주세요"
- PM: "일정 상 이 기능을 축소하는 게 좋겠습니다. 동의하시겠습니까?"

**검토 요청 (Review)**
- FE/BE: "이 코드가 작성되었습니다. 확인해주세요"
- Pln: "기획서 초안입니다. 검토해주세요"
- Dsn: "UI 시안입니다. 피드백 주세요"

**알림 (Notification)**
- PM: "티켓 A가 완료되었습니다"
- BM: "이번 티켓 예상 비용이 설정 한도를 넘습니다"
- GM: "외부 라이브러리 업데이트가 감지되었습니다"

## 9. HITL 담당자 매칭 (HR + BM)

### 9.1 매 HITL 노드마다 최적 담당자 제안

```
Agent가 HITL 필요 판단
    ↓
HR이 해당 HITL의 요구사항 분석
(필요 전문성, 난이도, 맥락 이해도, 예상 소요 시간)
    ↓
HR이 내부 풀 스캔
(팀원별 현재 일정, 전문성 매칭도, 해당 프로젝트 맥락 이해도)
    ↓
HR이 외부 풀 스캔 (GRIDGE 전문가 풀)
(전문성 매칭도, 시간당 비용, 가용 시간)
    ↓
BM이 ROI 비교 산출 (내부 vs 외부)
    ↓
PM이 최종 제안 구성 → 큐에 도착
```

### 9.2 ROI 비교 예시

```
── 내부 후보 ──
1. 김주니어 (L1) — 매칭도 72%, 오늘 오후 가용, 비용 $0
   ⚠️ 파일 업로드 경험 없음, 맥락 이해도 낮음, 재작업 확률 40%
2. 이시니어 (L2) — 매칭도 95%, 내일 오전 가용, 비용 $0
   현재 다른 티켓 리뷰 중

── 외부 후보 (GRIDGE 풀) ──
3. 박프론트 (React 전문) — 매칭도 91%, 즉시 가용, $45/hr
   유사 프로젝트 3건 경험

HR 추천: 긴급도에 따라 달라짐
  → 일정 여유 있으면: 이시니어 (내일)
  → 긴급하면: 박프론트 (즉시)
```

### 9.3 담당자 교체 트리거
- 현재 담당자 일정 충돌 → HR 감지 → 대체 인력 제안
- 에스컬레이션 발생 → 더 높은 전문성 필요 → HR 재매칭
- 스펙 변경 → 새로운 전문성 필요 → HR 재매칭
- 외부 전문가 세션 종료 → 후속 작업 내부 전환 판단

매 교체 시 **맥락 핸드오프**: GM이 의사결정 히스토리 + 코드 변경사항 정리 → 새 담당자 브리핑.

## 10. 에스컬레이션 / 디에스컬레이션

### 10.1 구조
HITL 과정에서 추가 예산, 더 높은 전문성, 상위 의사결정이 필요하면 단계별 상승. 반대로 실행 지시가 아래로 이동 가능.

**핵심: 맥락이 누적되면서 이동.** 상위 관리자가 받았을 때 처음부터 파악할 필요 없이, 하위에서 올라오며 쌓인 판단 근거를 바로 확인 가능.

### 10.2 권한 레이어

| 레이어 | 역할 예시 | 큐에 오는 것 | 에스컬레이션 |
|--------|---------|-------------|-------------|
| L1 | 주니어 개발자 | 코드 테스트, 단순 리뷰 | 위로만 |
| L2 | 시니어 개발자 | 코드 리뷰, 기술 스펙, L1 에스컬레이션 | 위/아래 |
| L3 | 팀 리더/CTO | 의사결정, 예산, 보안, L2 에스컬레이션 | 위/아래 |
| L4 | 대표/경영진 | 대규모 예산, 전략적 방향 | 아래로만 |

**레이어 수는 고정이 아님** — 고객사가 Settings에서 자유롭게 구성. PM Agent가 해당 설정 기반으로 자동 라우팅 + 에스컬레이션 판단.

### 10.3 에스컬레이션 시나리오 예시

```
L1(김주니어)이 코드 리뷰 중 충돌 발견
    ↓ 에스컬레이션 (코멘트 + 충돌 내용 첨부)
L2(이시니어) 큐에 도착: L1 코멘트 + 원본 코드 + 충돌 모듈 diff
    ↓ "리팩토링 필요, 추가 공수 발생" 판단 → 에스컬레이션
L3(김CTO) 큐에 도착: L2 판단 + L1 코멘트 + BM 추가 비용 + PM 일정 영향
    ↓ 승인 → PM 서브 티켓 자동 생성 → Flowchart 새 노드
    ↓ 디에스컬레이션 (실행 지시가 아래로)
L2 큐에: "승인됨, 이 방향으로 진행"
```

## 11. Flowchart 노드 생명주기

### 11.1 노드 타입 (5종)

| 타입 | 설명 | 상태 |
|------|------|------|
| Task (작업) | Agent 실행 단위. 서브 티켓 1:1 매핑 | 대기 → 진행 중 → 완료 |
| HITL (Human Decision) | 사람 판단 필요 지점. 담당자 표시 | 대기 → 진행 중 → 승인/반려 |
| Branch (분기) | 여러 경로 중 선택. HITL과 결합 빈번 | 선택/비활성 |
| Merge (병합) | 병렬 작업 합류. 모든 선행 노드 완료 시 활성화 | 대기 → 활성 |
| Escalation | 하위→상위 이동 지점. 맥락 누적 이력 포함 | 에스컬레이션됨 |

### 11.2 노드 상태별 시각적 차이
- waiting: 회색, 불투명 50%
- in_progress: 파란색, 펄스 애니메이션
- done/approved: 초록색, 체크마크 오버레이
- rejected: 빨간색, 이전 노드로 회귀 간선 생성
- escalated: 주황색, 새 HITL 노드 동적 삽입

### 11.3 동적 변경 케이스
- **노드 추가**: PM/Agent 서브 티켓 발행 또는 유저 직접 추가 (AI 소통 모달)
- **노드 삭제**: PM 의존성 영향 분석 → 유저 승인 → 간선 재연결
- **노드 순서 변경**: PM Flowchart 간선 재배치 + 영향 분석
- **노드 분열**: 작업이 예상보다 커진 경우, 하나가 여러 서브 노드로

### 11.4 유저 직접 편집 

**STEP 1**: 유저 변경 의도 → PM 판단 + 대안 제시 → 유저 수용 시 반영
**STEP 2**: PM 의견에 동의 못함 → 신규 채팅 모달에서 PM과 1:1 논의 → PM 설득되면 유저 방식 반영
**STEP 3**: 그래도 직접 수정 → Flowchart 직접 편집 모드 → "유저 직접 수정" 태그 부착 (추후 맥락 추적 + PM 학습 데이터)

**원칙: 최종 권한은 항상 유저에게. PM은 블로커가 아니라 리스크 알림 역할.**

### 11.5 Flowchart 레벨
- **Epic Flowchart**: 하나의 Epic 안에서 Ticket + Sub-task 흐름
- **Project Flowchart**: Epic 간 관계를 보는 상위 Flowchart
- 줌 인/줌 아웃으로 전환

### 11.6 완료된 Flowchart
아카이브 형태로 보관 → Document Bundle이 됨. "이 기능은 이런 과정으로 만들어졌다"는 문서 역할.

## 13. 거버넌스 (DB 접근 관리)

### 13.1 DB 연결 방식
- **읽기 전용 레플리카 연결 (권장)**
- Connection string 직접 연결 (읽기 전용 권한만)
- DB 덤프 업로드 (스키마만 / 스키마+샘플 데이터)

### 13.2 데이터 3단 분류

| 분류 | 정의 | 예시 | 기본 노출 범위 |
|------|------|------|-------------|
| 메타데이터 | 구조 정보. 테이블명, 컬럼명, 타입, 관계 | users 테이블에 email 컬럼(varchar) 존재 | Agent 전체 열람 가능 |
| 일반 데이터 | 업무상 참조 필요하나 민감하지 않은 데이터 | 상품 카탈로그, 카테고리 코드, 설정값 | Agent + L1 이상 열람 |
| 로우 데이터 | 개인정보, 결제, 인증 등 민감 원본 데이터 | 유저 이메일, 전화번호, 결제 내역 | 기본 차단, L3+ GM 승인 시에만 |

### 13.3 분류 방식

**테이블 단위로 GM 일괄 자동 분류** + **예외 컬럼만 유저 개별 지정**

GM 자동 분류 기준:
- 컬럼명 패턴으로 추론 (email, phone, password → 로우)
- 테이블 성격 추론 (users, payments → 민감 / products, categories → 일반)
- FK 관계로 민감 테이블과 연결된 참조 테이블 감지

GM 판단 불가 → 유저에게 넘김:
- 비즈니스 맥락 필요 (예: "memo" 컬럼이 내부 메모인지 고객 메모인지)
- 커스텀 암호화된 컬럼
- 회사 고유 정책 데이터

### 13.4 접근 정책 매트릭스

| 레이어 | 메타데이터 | 일반 데이터 | 로우 데이터 |
|--------|----------|-----------|-----------|
| Agent | ✅ 허용 | ✅ 허용 | ❌ 차단 |
| L1 | ✅ 허용 | ✅ 허용 | ❌ 차단 |
| L2 | ✅ 허용 | ✅ 허용 | ⚠️ 요청 필요 |
| L3 | ✅ 허용 | ✅ 허용 | ✅ 허용 |
| L4 | ✅ 허용 | ✅ 허용 | ✅ 허용 |
| 외부전문가 | ✅ 허용 | ⚠️ 요청 필요 | ❌ 차단 |

### 13.5 런타임 접근 제어

```
Agent가 데이터 참조 필요
    ↓
GM이 정책 확인
    ├─ 메타/일반 → 자동 허용
    └─ 로우 → 차단 + 마스킹 샘플 대체 제공
         └─ 정말 필요한 경우 → GM이 HITL 큐에 승인 요청
              └─ L3 승인: "이 티켓 한정 임시 허용"
                   └─ 티켓 완료 시 임시 권한 자동 회수
```

### 13.6 접근 로그
관리자 확인 가능: 시간, 주체(Agent/Human/외부), 대상 데이터, 분류, 액션, 사유

## 14. Document / Skill 체계

### 14.1 Document (결과물 기록)
Flowchart 완료 시 자동 생성되는 산출물 묶음 (Document Bundle):

| 유형 | 생성 주체 | 내용 |
|------|----------|------|
| 기획서 | Pln | 요구사항, 정책, 와이어프레임 |
| 기술 스펙 | SM | 선택된 스펙, 대안 비교, 선택 이유 |
| API 문서 | FE/BE | 엔드포인트, 요청/응답 스키마 |
| 코드 변경 이력 | FE/BE | 변경 파일, diff 요약, 의존성 영향 |
| 디자인 산출물 | Dsn | UI 시안, 컴포넌트 정의 |
| 의사결정 기록 | PM | HITL에서 유저가 내린 결정 + 근거 |
| 비용 리포트 | BM | 실제 소요 비용, 예산 대비 |
| 보안 검토 기록 | GM | 접근 허용 자료, 보안 판단 근거 |

Document 접근 권한은 레이어별로 다름. 외부 전문가는 참여 분만 + 작업 완료 후 회수.

### 14.2 Skill (재사용 패턴)

| 유형 | 설명 | 활용 |
|------|------|------|
| 프롬프트 Skill | 효과적이었던 프롬프트 패턴 | Agent가 유사 작업 시 자동 참조 |
| 아키텍처 Skill | 반복 사용 가능 기술 패턴 | SM이 유사 스펙 제안 시 추천 |
| 프로세스 Skill | 효과적이었던 작업 흐름 | PM이 소통 토폴로지 설정 시 참고 |
| 정책 Skill | 반복 적용되는 비즈니스 규칙 | GM이 자동 적용 |

### 14.3 Skill 등록 경로
- **PM 제안**: 티켓 완료 후 재사용 가치 판단 → 유저 확인
- **유저 직접**: 작업 중 아무 시점에서 등록 → PM 사후 검증 (Context Layer 충돌, 중복 확인)
- 사용 실적이 쌓이면서 검증. 많이 쓰이고 효과적이면 상위 노출, 안 쓰이면 아카이브 제안.

### 14.4 유저 AI 활용 능력 모니터링
AiOPS 코칭/성숙도 통합. 단순 AI 도구 사용 패턴이 아니라 "Agent와 협업하는 능력" 측정:
- HITL 처리 현황 (정확도, 응답 시간, 에스컬레이션 비율)
- Skill 활용도 (등록된 Skill 대비 사용률)
- AI 성숙도 5단계 (탐색기 → 실험기 → 정착기 → 확장기 → 최적화기)
- 주간 코칭 카드 자동 생성

## 15. 루틴 업무

### 15.1 정의
반복 발생 업무를 정의하여 HITL 없이 지속 자동 실행. **HITL을 졸업하는 구조.**

### 15.2 생명주기

```
1) PM이 유사 티켓 N회 이상 반복 감지
2) 루틴 등록 제안 (트리거 조건, 실행 Agent, 자동화 범위, 예외 조건)
3) 유저 승인 → 루틴 활성화
4) 트리거 발생 → Agent 자동 실행 → 완료 → 기록
5) 예외 발생 시 HITL 복귀
```

### 15.3 자동화 수준 (유저 설정)

| 수준 | 설명 | HITL |
|------|------|------|
| 알림만 | 실행 자동, 결과만 알림 | 없음 |
| 결과 확인 | 결과를 큐에 보여줌, 문제 없으면 자동 통과 | 최소 |
| 승인 필요 | 실행 준비까지 자동, 최종 실행은 승인 | 매회 |

신뢰 쌓이면 점진적으로 수준 올림.

### 15.4 Flowchart 표현
순환 화살표 아이콘 + 마지막 실행 시간 + 다음 실행 예정. 완료되지 않고 계속 살아있는 Flowchart. 클릭 시 실행 히스토리 + 성공/실패.

### 15.5 Skill 연동
루틴은 Skill 참조해서 실행. Skill 업데이트 시 루틴 방식도 반영. 반대로 루틴 데이터에서 새 Skill 추출 가능.

## 16. 외부 업무 탭 + GRIDGE 전문가 풀

### 16.1 하나의 인터페이스에서 내부/외부 탭 전환
Icon Nav의 💼 탭 클릭 시 Sub Nav + Main + Chat 전체가 외부 업무 컨텍스트로 전환. 내부 탭 복귀 시 자기 회사 컨텍스트로.

### 16.2 유저 케이스 3종

| 케이스 | 설명 |
|--------|------|
| A. 순수 내부 | 자기 회사 업무만, 외부 탭 미사용 |
| B. 순수 외부 | 프리랜서가 GRIDGE 풀 업무만 수행 |
| C. 하이브리드 | 회사 소속이면서 여유 시간에 외부 업무도 수행 |

### 16.3 외부 업무 플로우

```
새 업무 제안 카드 도착 (푸시 알림)
    ↓
업무 개요 + 예상 보수 + 난이도 (프로젝트 상세 비공개)
    ↓
수락
    ↓
GM이 필요 Context만 선별 공개 (보안 등급에 따라)
    ↓
Main에 작업 화면 + Chat에 맥락 브리핑
    ↓
완료 제출
    ↓
내부 담당자(L2/L3) 검토 → 승인/반려
    ↓
승인 시 정산 + Context 접근 권한 회수
```

### 16.4 보안 경계 (GM 관할)

| 구분 | 내부 팀 | 외부 전문가 |
|------|---------|-----------|
| 프로젝트 목록 | 전체 | 할당된 티켓만 |
| 코드 접근 | 전체 레포 | GM 선별 모듈만 |
| Context Layer | 전체 | 해당 작업 관련만 |
| 작업 완료 후 | 유지 | 접근 권한 즉시 회수 |

### 16.5 관리자 설정 (Settings > 외부 업무 정책)
- 외부 업무 허용 여부 (ON/OFF)
- 허용 시간 제한 (주 N시간)
- 제한 조건 (경쟁사 불가, 특정 도메인 불가 등)
- 수익 귀속: 회사 / 개인 / 비율 분배
- 팀원별 개별 설정 가능

### 16.6 수익 모델
- 고객사 → Wiring 구독료
- 외부 전문가 매칭 → GRIDGE 중간 수수료
- 수익 회사 귀속 시 → 유휴 인력 수익화 채널
- 수익 개인 귀속 시 → 복리후생/인센티브 성격

기존 Gridge 전문가 매칭 사업이 Wiring 안으로 자연스럽게 흡수.
